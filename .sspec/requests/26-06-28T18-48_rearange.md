# RFC: `asq rearrange` — 基于行区间的文本块重排 CLI 设计规格

## 1. 文档定位

本文档是 `asq rearrange` 的 **Agentic Coding Implementation Spec**。

它用于在 agentic coding 过程中，将讨论后的实现目标、外部行为、输入输出契约、安全边界和核心内部机制固化下来，使后续编码 Agent 可以基于本文档实现功能，而不需要重新推断需求。

本文档不是传统 PRD。它不重点讨论市场价值、用户画像或商业目标，而是定义：

1. CLI 应该对外呈现什么行为。
2. 输入格式应该如何设计。
3. Agent 应该如何使用。
4. 哪些情况应该失败。
5. 写入、备份、回滚等内部机制应该满足什么边界。
6. 哪些复杂性必须由 CLI 内部吸收，而不是暴露给 Agent。

一句话目标：

> `asq rearrange` 是一个声明式 line-range chunk transaction CLI：用户或 Agent 只定义文本块与动作，CLI 内部负责解析、校验、diff、备份、事务式写入与失败恢复。

---

## 2. 背景与问题

在 Agent 修改本地文件时，常见需求包括：

1. 把一个连续文本块移动到文件其他位置。
2. 把多个连续文本块重新排序。
3. 把某个块从一个文件移动到另一个文件。
4. 删除某个块。
5. 复制某个块到另一个位置。
6. 在多文件场景下批量执行上述操作。

目前可以使用 `asq patch` 的 SEARCH/REPLACE 完成类似任务，但存在问题：

1. 大块移动时 SEARCH/REPLACE 内容冗长。
2. Agent 容易因为复制原文不完整导致匹配失败。
3. 通过 patch 表达“移动”时，本质上需要同时写删除和插入，语义不直接。
4. 多块重排时 patch 容易发生重叠、顺序污染和误删。
5. Agent 应该只声明“我要移动哪些行”，不应该手动处理底层 hash、换行、备份、回滚等细节。

因此需要一个专门面向文本块重排的 CLI。

---

## 3. 产品定位

`asq rearrange` 负责：

```text
基于 1-based line range 定义文本块，并对这些文本块执行 move / copy / delete / rearrange。
```

它和现有命令的分工：

| 命令               | 职责                                        |
| ---------------- | ----------------------------------------- |
| `asq read-range` | 读取指定行区间，为 Agent 提供行号依据                    |
| `asq patch`      | 修改文本内容，适合 SEARCH/REPLACE、CREATE、OVERWRITE |
| `asq rearrange`  | 移动、复制、删除、重排连续文本块                          |

`asq rearrange` 的输入应该简单：

```text
chunks + actions
```

其中：

```text
chunks  = 定义哪些连续行区间是可操作文本块
actions = 定义对这些块执行什么操作
```

Agent 不需要，也不应该，提供：

```text
hash
newline
encoding
BOM
backup path
journal id
fsync 策略
rollback 状态
```

这些全部属于 CLI 内部职责。

---

## 4. 非目标

`asq rearrange` v1 不做：

1. 不做 Markdown heading selector。
2. 不做 AST symbol selector。
3. 不做 regex block selector。
4. 不做自然语言解析。
5. 不自动格式化内容。
6. 不自动修复缩进。
7. 不改写 chunk 内部文本。
8. 不创建缺失目标文件。
9. 不要求 Agent 提供 hash、newline、encoding、backup 信息。
10. 不承诺数据库级跨文件原子事务。

高级 selector 可以未来通过 `resolve` 类命令实现，例如：

```text
heading/regex/symbol -> line range
```

但 `rearrange` 核心语义只接受 line range。

---

## 5. 核心概念

### 5.1 Chunk

Chunk 是基本操作对象。

```text
chunk = file + 1-based inclusive line range
```

例如：

```text
chunk A = README.md:L10-L30
```

含义：

```text
A 是 README.md 中第 10 行到第 30 行，包含第 10 行和第 30 行。
```

### 5.2 Action

Action 是对 chunk 的操作。

v1 支持：

```text
move
copy
delete
rearrange
```

### 5.3 Anchor

Anchor 是插入位置。

v1 支持：

```text
start
end
before L<N>
after L<N>
```

示例：

```text
after L120
```

含义：

```text
插入到原始文件第 120 行之后。
```

### 5.4 原始快照坐标系

所有 chunk range 和 anchor 都基于 **原始文件快照** 解释。

例如：

```text
chunk A = L10-L20
move A to after L50
```

无论删除 `L10-L20` 后行号如何变化，`after L50` 都指的是操作开始前原始文件的第 50 行。

这条规则必须固定，否则多 action 场景会出现行号污染。

## 6. CLI 入口

建议命令：

```bash
asq rearrange [SPEC]
```

建议选项：

```text
--cwd <DIR>               在指定目录下执行
-f, --file <PATH>         从文件读取 spec
--stdin                   从 stdin 读取 spec
--dry-run                 只校验和输出 diff，不写入
-y, --yes                 执行写入；没有 --yes 时不得修改文件
--print <MODE>            compact | json | ndjson | text | raw
--json                    等价于 --print json
--prompt                  输出格式说明
--no-backup               禁用自动备份；不建议 Agent 使用
--list-backups            列出最近 backup transaction
--rollback <TXN_ID>       从 backup 回滚
--recover                 检查并处理未完成 transaction
```

默认行为：

```text
默认 dry-run。
写入必须显式传入 --yes。
写入默认自动 backup。
```

推荐 Agent 用法：

```bash
asq rearrange --stdin --dry-run < ops.rearrange
asq rearrange --stdin --yes < ops.rearrange
```

```bash

```

## 8. JSON Schema v1

### 8.1 顶层结构

```json
{
  "version": 1,
  "chunks": {
    "<chunk_name>": {
      "file": "<path>",
      "range": "L<start>-L<end>"
    }
  },
  "actions": []
}
```

### 8.2 Chunk 定义

```json
{
  "file": "README.md",
  "range": "L10-L30"
}
```

规则：

```text
range 使用 1-based inclusive line range。
v1 支持 L10 和 L10-L30。
v1 不支持 L10-、-L30、start-end 等开放区间。
```

### 8.3 Action: move

```json
{
  "type": "move",
  "chunk": "A",
  "to": {
    "file": "README.md",
    "anchor": "after L120"
  }
}
```

语义：

```text
从原位置剪切 chunk A，并插入到目标 anchor。
```

### 8.4 Action: copy

```json
{
  "type": "copy",
  "chunk": "A",
  "to": {
    "file": "README.md",
    "anchor": "end"
  }
}
```

语义：

```text
复制 chunk A 到目标 anchor，原位置保留。
```

### 8.5 Action: delete

```json
{
  "type": "delete",
  "chunk": "A"
}
```

语义：

```text
删除 chunk A。
```

### 8.6 Action: rearrange

```json
{
  "type": "rearrange",
  "file": "README.md",
  "from": ["A", "B", "C", "D"],
  "to": ["B", "D", "C", "A"],
  "gap": "slot"
}
```

语义：

```text
在同一文件内，将 from 中声明的 chunk 槽位按 to 的顺序重新填充。
```

约束：

```text
from 和 to 必须包含完全相同的 chunk 集合。
这些 chunks 必须属于同一个 file。
chunks 之间不能重叠。
默认 gap=slot。
```

## 10. Gap 语义

### 10.1 问题

`rearrange` 允许声明不连续 chunk。

例如原文件局部结构：

```text
A, hidden1, B, C, hidden2, D
```

用户声明：

```text
rearrange A, B, C, D => B, D, C, A
```

需要明确 `hidden1` 和 `hidden2` 如何处理。

### 10.2 默认策略：gap=slot

默认：

```text
gap=slot
```

语义：

> hidden lines between declared chunks are treated as inter-slot gaps. Rearranging chunks replaces chunk slots but keeps hidden gaps in their original inter-slot positions.

中文定义：

> 声明 chunk 之间的未定义行被视为槽位间隔。重排时只替换 chunk 槽位，hidden gap 保持在原始槽位之间。

示例：

原始：

```text
A, hidden1, B, C, hidden2, D
```

操作：

```text
A, B, C, D => B, D, C, A
```

结果：

```text
B, hidden1, D, C, hidden2, A
```

内部模型：

```text
slot1=A, gap1=hidden1, slot2=B, gap2=empty, slot3=C, gap3=hidden2, slot4=D
```

重排后：

```text
slot1=B, gap1=hidden1, slot2=D, gap2=empty, slot3=C, gap3=hidden2, slot4=A
```

### 10.3 gap=drop

```text
gap=drop
```

语义：

```text
只保留声明的 chunks，丢弃所有未定义 gap。
```

示例：

```text
A, hidden1, B, C, hidden2, D
A, B, C, D => B, D, C, A gap=drop
```

结果：

```text
B, D, C, A
```

dry-run 必须明确提示被丢弃的 gap 行区间。

### 10.4 gap=error

```text
gap=error
```

语义：

```text
如果 declared chunks 之间存在未定义行，则失败。
```

适合需要极高保守性的场景。

### 10.5 v1 gap policy

v1 支持：

```text
slot
drop
error
```

默认：

```text
slot
```

## 11. 外部行为定义

### 11.1 dry-run

没有 `--yes` 时，工具不得写入文件。

dry-run 应执行：

```text
1. 解析 spec。
2. 读取相关文件。
3. 解析 chunks 和 anchors。
4. 校验所有 actions。
5. 在内存中生成新内容。
6. 输出 summary / diff / json。
```

dry-run 不创建 backup。

### 11.2 写入

传入 `--yes` 后才允许写入。

写入前仍需完整执行：

```text
parse -> resolve -> validate -> materialize -> snapshot check -> backup -> write -> verify
```

### 11.3 compact 输出示例

```text
REARRANGE README.md

Chunks:
  A: L1-L30       30 lines
  B: L35-L50      16 lines
  C: L51-L80      30 lines
  D: L90-L120     31 lines

Actions:
  rearrange A, B, C, D => B, D, C, A
  gap: slot

Gaps:
  G1: L31-L34 kept between slot1 and slot2
  G2: L81-L89 kept between slot3 and slot4

Files:
  README.md modified

Dry-run only. Use --yes to apply.
```

## 13. 安全机制

### 13.1 Snapshot check

Agent 不提供 hash。

CLI 内部负责：

```text
1. 读取目标文件并记录内部快照。
2. 生成新内容。
3. 写入前再次读取目标文件。
4. 如果文件已变化，拒绝写入。
```

对外错误：

```text
FILE_CHANGED: README.md changed during planning/apply. No files were written.
```

### 13.2 Newline / encoding preservation

Agent 不提供 newline、encoding、BOM。

CLI 内部负责：

```text
1. 检测原文件 newline style。
2. 检测 BOM / encoding。
3. 写回时保持目标文件风格。
4. 跨文件 move/copy 时，插入内容转换为目标文件 newline style。
5. 无法安全处理编码时失败。
```

### 13.3 Auto backup

写入时默认自动备份所有将被修改的文件。

备份位置建议：

```text
.asq/backups/rearrange/<txn-id>/
```

示例：

```text
.asq/backups/rearrange/20260628-171612-a8f31c/
  README.md
  docs/api.md
  journal.json
```

`journal.json` 记录：

```text
transaction id
command
status
original file path
backup file path
write state
rollback state
```

### 13.4 Best-effort transaction

多文件写入不能承诺数据库级原子性。

对外承诺：

```text
pre-validate all changes
auto-backup modified files
write through temp files
rollback from backups on failure
report manual recovery path if rollback fails
```

文档表述：

```text
asq rearrange provides best-effort transactional apply with automatic backup and rollback. It does not provide database-grade atomicity across arbitrary file systems.
```

## 14. 内部执行流程

### Phase 1: Parse

输入：

```text
DSL 或 JSON
```

输出：

```text
统一内部 Spec
```

检查：

```text
version 是否支持
chunks 是否存在
actions 是否合法
chunk name 是否重复
```

### Phase 2: Resolve

读取涉及文件。

解析：

```text
chunk range
target anchor
file paths
```

内部记录：

```text
文件快照
行数组
newline style
encoding/BOM
```

这些不属于 public input schema。

### Phase 3: Validate

校验：

```text
range 是否越界
source chunks 是否重叠
action 是否引用未知 chunk
move/delete anchor 是否落在被移动 chunk 内
rearrange from/to 是否同集合
rearrange 是否跨文件
target file 是否存在
目标文件是否可写
```

### Phase 4: Materialize

在内存中生成每个目标文件的新内容。

关键规则：

```text
所有 source range 和 anchor 都基于原始快照。
同一文件多操作不得按顺序污染行号。
最终写入内容由统一 planner 一次性计算。
```

### Phase 5: Diff

生成 compact summary 和 unified diff。

dry-run 在此结束。

### Phase 6: Pre-commit snapshot check

写入前重新读取目标文件，确认与内部快照一致。

不一致则失败：

```text
FILE_CHANGED
```

### Phase 7: Backup

为所有将被修改文件创建备份。

写入 journal：

```text
status = preparing / committing / committed / rollback
```

### Phase 8: Write

对每个文件：

```text
write temp file in same directory
replace original file
update journal state
verify new content
```

### Phase 9: Rollback

若写入失败：

```text
从 backup 恢复已写入文件
更新 journal
输出 rollback 结果
```

## 15. 同文件多 action 规则

所有 action 基于原始文件坐标解释。

v1 保守规则：

```text
同一文件 source chunks 不能重叠。
target anchor 不能位于将被 move/delete 的 chunk 内。
delete chunk 不能被后续 action 引用为 source。
同一 chunk 在 actions 中只能作为消耗性 source 出现一次。
copy 不消耗 chunk。
move/delete/rearrange 会消耗 chunk。
```

原因：

```text
避免多 action 中出现隐式二次移动、重复删除、锚点漂移等复杂语义。
```

## 16. 文件路径规则

路径解析：

```text
相对路径基于 --cwd。
```

v1 建议：

```text
允许普通相对路径和绝对路径。
目标文件不存在时默认失败。
备份时保留相对路径结构。
```

暂不做：

```text
--create-missing
路径 sandbox 限制
symlink 策略细分
```

后续可以增加。

## 17. UX 成功标准

### 17.1 对 Agent

Agent 应该只需要做到：

```text
1. 用 read-range / gather / editor 上下文拿到行号。
2. 定义 chunks。
3. 写 actions。
4. dry-run。
5. 根据错误修正。
6. --yes 应用。
```

Agent 不需要知道：

```text
hash
newline
encoding
backup
journal
rollback
fsync
```

### 17.2 对人类

人类应能快速理解：

```text
先 chunk，后 action。
行号是 1-based inclusive。
rearrange 默认保留 hidden gaps 在 slot 位置。
默认 dry-run。
--yes 才写。
```

## 18. `--prompt` 推荐内容

````markdown
# Squire rearrange format

`asq rearrange` moves, copies, deletes, and reorders line-range chunks.

## Model

A spec has two parts:

1. Define chunks.
2. Execute actions.

Ranges are 1-based and inclusive: `L10-L20` includes both line 10 and line 20.

All chunks and anchors are resolved against the original file snapshot.

## DSL

```text
file <path>

chunk <name> = L<start>-L<end>
chunk <name> = <path>:L<start>-L<end>

move <chunk> to <target>
copy <chunk> to <target>
delete <chunk>

rearrange <chunk-list> => <chunk-list> [gap=slot|drop|error]
````

## Targets

```text
start
end
before L<N>
after L<N>
<path>:start
<path>:end
<path>:before L<N>
<path>:after L<N>
```

## Gap

Default: `gap=slot`.

Hidden lines between declared chunks are treated as inter-slot gaps. Rearranging chunks replaces chunk slots but keeps hidden gaps in their original inter-slot positions.

Example:

```text
A, hidden1, B, C, hidden2, D
A, B, C, D => B, D, C, A
```

Result:

```text
B, hidden1, D, C, hidden2, A
```

## Example

```text
file README.md

chunk A = L1-L30
chunk B = L31-L50
chunk C = L51-L200

rearrange A, B, C => B, C, A
```

## JSON

```json
{
  "version": 1,
  "chunks": {
    "A": {"file": "README.md", "range": "L1-L30"},
    "B": {"file": "README.md", "range": "L31-L50"}
  },
  "actions": [
    {
      "type": "rearrange",
      "file": "README.md",
      "from": ["A", "B"],
      "to": ["B", "A"],
      "gap": "slot"
    }
  ]
}
```

## Safety

* Default mode is dry-run.
* Use `--yes` to write.
* Writes create automatic backups.
* Same-file chunks must not overlap.
* Multi-file writes use best-effort transaction with rollback.


## 21. 验收用例

### Case 1: 单文件 move

输入：

```text
file a.md
chunk A = L10-L20
move A to after L50
```

期望：

```text
L10-L20 被剪切并插入到原始 L50 后。
默认 dry-run 不写文件。
--yes 写入并备份。
```

### Case 2: 单文件 rearrange with continuous chunks

输入：

```text
file a.md
chunk A = L1-L10
chunk B = L11-L20
chunk C = L21-L30
rearrange A, B, C => C, A, B
```

期望：

```text
文件前 30 行顺序变成 C/A/B。
```

### Case 3: rearrange with gap=slot

原始结构：

```text
A, hidden1, B, C, hidden2, D
```

输入：

```text
file a.md
chunk A = L1-L10
chunk B = L15-L20
chunk C = L21-L30
chunk D = L40-L50

rearrange A, B, C, D => B, D, C, A
```

期望：

```text
结果为 B, hidden1, D, C, hidden2, A。
hidden gaps 保留在原始 slot 间位置。
```

### Case 4: rearrange with gap=drop

输入：

```text
file a.md
chunk A = L1-L10
chunk B = L15-L20

rearrange A, B => B, A gap=drop
```

期望：

```text
L11-L14 被丢弃。
dry-run 必须明确提示 dropped gap。
```

### Case 5: overlapping chunks

输入：

```text
file a.md
chunk A = L10-L20
chunk B = L15-L30
rearrange A, B => B, A
```

期望：

```text
失败：OVERLAPPING_CHUNKS。
```

### Case 6: cross-file move

输入：

```text
chunk API = README.md:L40-L90
move API to docs/api.md:end
```

期望：

```text
README.md 删除 API chunk。
docs/api.md 末尾追加 API chunk。
两文件均备份。
任一写入失败则回滚。
```

### Case 7: file changed

流程：

```text
dry-run 后，用户修改了 README.md，再执行 --yes。
```

期望：

```text
失败：FILE_CHANGED。
不写文件。
```

## 22. 最终外部承诺

`asq rearrange` 对外承诺：

```text
1. 输入简单：只需要 chunks + actions。
2. 坐标明确：1-based inclusive line range。
3. 重排明确：gap 默认使用 slot 语义。
4. 默认安全：dry-run 默认，--yes 才写。
5. 操作可见：写入前可看到 summary/diff/json。
6. 写入可恢复：默认自动 backup。
7. 失败可解释：结构化错误码。
8. 内部复杂性不泄漏：Agent 不写 hash/newline/encoding/journal。
```

最终定义：

> `asq rearrange` 是一个为 Agentic Coding 场景设计的文本块重排命令。它使用简单的 `chunks + actions` 输入模型，让 Agent 通过 1-based line range 声明文件块操作；CLI 内部负责解析、校验、gap 处理、diff、自动备份、事务式写入与失败恢复。
